# PM 提案生成模板（既有產品 · 用新功能強化北極星 · 套用即生成）

> **這份文件本身就是一個 prompt。** 把你的 `spec.md` + `plan.md` 貼進下方 `<輸入文件>`，連同整份模板一起丟給任何 LLM（Claude / GPT…），它會吐出一份**可直接貼進簡報工具的提案內容**。

---

## 適用情境（先確認用對版本）

這一版主線是「**指標貫穿全場**」，從北極星開場 —— 它**預設你已經有產品在跑、手上有真實使用數據**，故事是「既有產品 → 用新功能強化既有北極星」。

| 你的情況 | 用哪一版 |
|---|---|
| **已有產品**、要用新功能強化既有北極星 | ✅ 本版（`proposal-template.md`） |
| **全新產品**、提案當下還沒有產品 / 沒有數據 | ➡️ 改用 `proposal-template-new-product.md`（問題優先；北極星是「上線後承諾兌現的目標」） |

> 用錯版本會讓故事線接不起來：新產品還沒有數據，卻被北極星開場，聽眾無從理解。不確定就先看上表。

## 使用方式

1. 把你的 `spec.md` 與 `plan.md` 全文貼到下面 `<輸入文件>` 之間。
2. 把整份文件（含本說明、生成指令、STEP 區塊、附錄）一次丟給 LLM。
3. LLM 會逐一產出各頁內容；抽不到的資訊會標 `【待確認：…】`（可推斷）或 `【需補充：…】`（無依據）。
4. 把產出貼進你的簡報工具 → 生成簡報。

```
<輸入文件>
（把你的 spec.md 全文貼這裡）

---

（把你的 plan.md 全文貼這裡）
</輸入文件>
```

## 生成指令（給 LLM）

你是一位資深 PM 簡報教練。請**只根據 `<輸入文件>` 的內容**，為下方 7 個 STEP 逐一產出提案逐頁內容。**這是要給人看、要上台講的簡報，不是內部文件。** 嚴格遵守以下規則：

### A. 主線：指標貫穿全場（既有產品）

- 故事線：**北極星（產品靠什麼指標活）→ 那指標生什麼病 → 你的功能怎麼治 → 用功能指標證明療效。** 7 頁要能串成這條線，**每一頁都要扣回北極星**，不能各說各話。
- 既有產品才適用本主線（你有數據佐證北極星現況）。若其實是新產品，改用新產品版。

### B. 北極星規則（先讀附錄 A）

- 北極星要量「**使用者有沒有真的感受到價值**」（需求面 / 被採納），不能只量「系統做了什麼」（供給面）。
- 它必須**算得出來**：有使用者識別才用「使用者數」；若無識別 / 每個 session 行為量不一，改用**事件/產出單位的全域加總**（見附錄 A）。
- 門檻不能太鬆（避免「每人 ≥1 次」打勾就過）；量（count）要會成長，再配一個比率（rate）當護欄。

### C. 指標體系規則（漏斗 + 護欄，先讀附錄 B）

- 指標串成漏斗：`原始量能 ×（功能層轉換率）＝ 北極星（價值已實現的量）`，各層**不要撞名 / 語意重疊**。
- 護欄分兩類：**正確性**（品質對不對）與**行為健康**（行為健不健康）；都用**可量測可優化的比率**、方向一致。
- **不要**用絕對門檻（如「= 0」無法迭代）、**不要**用高變異難監測的量（如平均 session 時長）、**不要**把已被技術解決的問題當 KPI。

### D. 簡報語言規則（最容易出錯，務必遵守）

- **每頁都要輸出一個「白話標題」（會印在投影片上）**，講這頁在說什麼，**不要用內部術語當標題**。
  - ❌「機制 — 功能怎麼推動北極星」　✅「產品怎麼運作 — 從 X 到 Y」
- **「STEP N」只是這份模板的內部生成順序標記，絕對不要印在投影片上。**
- **專有名詞（北極星、護欄、轉換率…）在第一次出現的那頁先用一句話定義，再使用。** 不要假設聽眾懂。
- **不要赤裸地展現框架感**：不要做一頁「框架總覽」、不要把流程標籤直接當內容。

### E. 視覺與排版建議（給後續排版 / 生成 SVG 用）

- **證據用大數字當主角**，不要堆成一段論文。任一區塊**項目 ≤ 3 條**；長統計改「一句重點 +（來源）」。
- **每頁一條清楚的閱讀順序**（由上而下分層標題、或由左到右），別讓兩欄並列互搶、沒有主次。
- **留足呼吸感**：寧可少寫，不要塞滿。

### F. 誠實規則

- **數字只在 STEP 7（驗證）與必要處出現；STEP 4（成功定義）不放數字。**
- 沒有真實數據標 `【需補充】`；能依產品本質合理推斷標 `【待確認：依據】`；**一律不要編造。**
- 任何門檻 / 區間數值都註明「**假設，上線後用真實數據校準**」。

### G. 抽取三層規則

(1) **找得到** → 直接用；(2) **找不到、但能依產品本質合理推斷**（常見於北極星、使用者、指標）→ 草擬並標 `【待確認：依據】`；(3) **完全沒依據**（真實訪談、實際流量）→ 標 `【需補充】`。

### H. 數據驗證與並行搜尋規則（適用於具備網路搜尋能力之 Agent）

若執行本模板的 AI Agent 具備工具調用或網路搜尋能力，在產出 `STEP 2 · 可優化處` 頁面（簡報第一頁 Problem Exploration）時，**必須**強制執行以下元指令（meta-instructions）：
1. **分析提取痛點**：從 `<輸入文件>` 的 `spec.md` 中提取核心痛點與業務領域。
2. **設計 10 個搜尋查詢**：基於提取的核心痛點，設計 10 個具體的網路搜尋查詢（Search Queries），重點聚焦於市場規模、痛點工時、財務損失、錯誤率、行業調查或權威報告。
3. **並行啟動 10 個搜尋子代理人**：調用搜尋工具或並行啟動 10 個搜尋子代理人（parallel search subagents），分別執行上述查詢，蒐集具體定量統計、權威機構名稱（如 McKinsey、Bloomberg、WBCSD、Asana 等）與精確網址（URL）。
4. **融入實證數據至 Slide 1**：將蒐集到的真實數據與來源連結融入到 `STEP 2` 的「現象」中，完全替換任何推測或 `【需補充】` 標記。

### I. 兩個常踩的內容坑

- **機制要講「為什麼能帶動」，不要只列功能。**「SSE 串流 + 表格」不是理由；「因為快速易懂、降低理解成本，使用者才願意多用」才是。
- **解方頁放「有實際內容」的畫面**（真實問答 / 結果），不要只放空的入口頁 / 登入頁。

### J. 主產品定位與共識規則（適用於既有產品功能更新提案）

在產出 `STEP 1 · 北極星` 頁面（簡報第一頁 Current Status）前，AI Agent **必須**執行以下元指令（meta-instructions）：
1. **盤點主產品**：從 `<輸入文件>` 的 `spec.md` 及 `plan.md` 中盤點出該功能更新附屬的主產品為何。
2. **啟動 5 隻 Agent 共識會議**：若從輸入文件中難以辨識或定義主產品，必須虛擬啟動 5 隻 Agent 進行定位共識會議，定調主產品的定位、核心功能與大盤目標。
3. **競品搜尋與對齊**：針對已達成共識的主產品，進行國內外近似產品的 Web Search，以徹底了解其標準產品規格與特點，並以此共識來定位提案。
4. **輸出警告標記**：在最終輸出的簡報內容與演講講稿中，只要提及該主產品名稱或其指標大盤數據，必須在旁加上顯著的警告標籤：`[⚠️ 請在此處更新您正確的主產品名稱及指標]`，提醒使用者必須自行確認與修改為真實的主產品資訊。

> 輸出語言：繁體中文。few-shot 以「104 升學面試顧問」當**示範**（學它的具體程度與語氣，不要照抄內容）。

## STEP 1 · 北極星（內部標記；投影片標題請白話）

- **核心問句**：這個產品靠什麼單一指標衡量「核心價值有沒有被兌現」？為什麼是它？
- **填空骨架**：
  - 北極星指標是：______（單一、跨團隊共享、領先而非落後；先讀附錄 A）
  - 由這些要件構成：______（量的條件 + 質/投入的條件）
  - 為什麼不是註冊數/營收：______
- **抽取規則**：到 spec 找「核心價值 / 目標 / 願景」；plan 若有指標一併看。把「價值主張」翻成一個**可量測、能反映價值被感受到**的領先指標。
- **104 few-shot**：
  > 北極星 =「單月進站 ≥3 次 ＆ session 時長中位數 >10 分」的使用者數。
  > 為什麼：註冊是漏斗早期、營收是落後指標；此定義同抓回訪頻率（量）與投入深度（質）。
- **輸出格式**：
  ```

  - 北極星 ＝【指標】（單一、領先）
  - 構成要件：量【…】＋ 質【…】
  - 為什麼是它（非註冊/營收）：【…】
  ```

- **核心問句**：北極星哪裡出問題？用什麼證據？為什麼值得解？

## 附錄 A · 北極星設計

**好北極星三條件**：

1. **量「價值被感受到」**（需求面 / 被採納），不是「系統做了什麼」（供給面）。
2. **算得出來。**
   - **有使用者識別**（登入）→ 可用「使用者數」（如月活、回訪使用者數）。
   - **無使用者識別 / 每個 session 行為量不一** → 不要用「使用者數」或「每 session 平均」；改用**事件/產出單位的全域加總**（一則回答、一次完成…當計數單位，整段期間加總）。一個 session 做 10 次就貢獻 10 個、做 1 次就 1 個，**長 session 不會灌歪**。
3. **門檻不能太鬆。** 別用「每人 ≥1 次」打勾就過；用**會成長的量（count）**當北極星，再搭**同行為的比率（rate）當護欄**（量要長、率別崩）。

**命名公式**：`每週/月「被 <價值動作> 的有效 <產出單位> 數」`（例：被查核的有效回答數、被完成的有效任務數）。

## 附錄 B · 指標體系與護欄原則

**1. 漏斗串得起來、別撞名**

```
原始量能（總操作數）
   ×  功能層轉換率（變成「有效」的比率）
   =  北極星（價值已實現的量）
```
北極星=量（count，要成長）；功能層=率（轉換率）；量能=原始活躍量。三者語意不重疊（北極星叫「有效 X 數」，量能就別叫「有效 X 數」，改叫「總 X 數」）。

**2. 護欄分兩類、都要可量測可優化**

| 類別 | 管什麼 | 例 | 方向 |
|---|---|---|---|
| 正確性 | 結果對不對 | 覆蓋率、忠實度、回饋有效率 | 不下降 |
| 行為健康 | 行為健不健康 | 有效互動率、來源查核率 | 不下滑 |

**3. 護欄「不要」清單（實戰踩過的坑）**

- ❌ **絕對門檻（如「= 0」）** → 無法量測表現、無法迭代。改成**比率不下降**。
- ❌ **平均 session 時長** → 高變異、被長 session 當離群值、要定義 session 邊界。改成**比率型**。
- ❌ **「任何點擊都算」的太鬆指標** → 好奇亂點就達標。改成**有語意的動作**。
- ❌ **需要使用者識別卻沒有識別的指標** → 算不出來。改成**事件/產出層級**。
- ❌ **已被技術解決的問題（如「截斷率」）** → 不是 KPI，別放護欄。
