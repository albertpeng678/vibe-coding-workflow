# 新產品提案生成模板（無既有產品 · 問題優先 · 套用即生成）

> **這份文件本身就是一個 prompt。** 把你的 `spec.md` + `plan.md` 貼進下方 `<輸入文件>`，連同整份模板一起丟給任何 LLM（Claude / GPT…），它會吐出一份**問題優先、可直接貼進簡報工具的新產品提案內容**。
>
> **這一版專給「全新產品提案」用 —— 提案當下還沒有產品、沒有真實使用數據。**

---

## 為什麼要分一版（先看這段，別用錯模板）

通用版 `proposal-template.md` 的主線是「**指標貫穿全場**」，從 **STEP 1 北極星** 開場 —— 它**預設你已經有產品在跑**，故事是「既有產品 → 用新功能強化既有北極星指標」。

但如果是**全新產品提案**（原本沒有產品），一開場就丟北極星指標會出問題：

- 你**還沒有產品、沒有任何真實使用數據**，把北極星當「現在量到的數字」呈現，會像個沒有根基的宣稱。
- 故事線**接不起來**：聽眾還不知道你要解決什麼問題、產品長什麼樣，就先被丟一個指標，無從理解。

**新產品提案的正確主線是「問題優先」**：先把問題講透，再說使用者、成功長什麼樣、產品怎麼解、怎麼運作，**最後**才談「上線後要用什麼指標證明它成功」。北極星是「**承諾要兌現的目標**」，不是「現在的量測」。

| | 通用版（`proposal-template.md`） | 本版（新產品） |
|---|---|---|
| 適用 | 已有產品、要強化既有北極星 | 全新產品、提案當下沒有產品/數據 |
| 主線 | 指標貫穿全場（北極星開場） | 問題優先（北極星延後、定義在使用處） |
| 北極星 | 現有量測 | 上線後承諾兌現的目標 |
| 數據 | 可引用真實流量 | 多為假設，標 `【需補充】`、上線校準 |

## 使用方式

1. 把你的 `spec.md` 與 `plan.md` 全文貼到下面 `<輸入文件>` 之間。
2. 把整份文件（含本說明、生成指令、STEP 區塊、附錄）一次丟給 LLM。
3. LLM 會逐一產出各 STEP 的逐頁內容；抽不到的資訊會標 `【需補充：…】` 或 `【待確認：…】`。
4. 把產出貼進你的簡報工具（Gamma / SVG 生成器…）→ 生成簡報。

```
<輸入文件>
（把你的 spec.md 全文貼這裡）

---

（把你的 plan.md 全文貼這裡）
</輸入文件>
```

## 生成指令（給 LLM）

你是一位資深 PM 簡報教練。請**只根據 `<輸入文件>` 的內容**，為下方各 STEP 逐一產出**新產品提案**的逐頁內容。嚴格遵守以下規則：

### A. 主線：問題優先（不要指標開場）

故事線固定為：**問題 → 誰在痛 → 怎樣算成功 → 我們的解方 → 它怎麼運作（此時才交代要兌現的目標指標）→ 怎麼驗證**。

- **第一頁就是「問題」，不是北極星。** 北極星指標**最早只能出現在「運作方式」那頁**（且要先定義、再使用），完整指標體系放在最後的「驗證計畫」。
- 全程把北極星講成「**上線後承諾要兌現的單一目標**」，不要寫成現在量到的數字。

### B. 北極星規則（新產品最關鍵，先讀附錄 A）

- 北極星要能衡量「**使用者有沒有真的感受到價值**」（需求面 / 被採納），不能只衡量「系統做了什麼」（供給面）。一個「系統單方面產出了東西、但沒人真的用」的指標太鬆。
- 它必須**在你目前的條件下算得出來**。常見限制與解法見**附錄 A**：
  - **不知道使用者是誰（無登入/無使用者識別）** → 不要用「使用者數」；改用**事件 / 產出單位**（一則回答、一筆結果）當計數單位。
  - **每個 session 的提問/操作數都不一樣** → 不要用「每 session 平均」（會被長 session 灌歪），改用**整段期間的全域加總**。
- 不要設「太容易達標」的門檻（例如「每人每週 ≥1 次」打勾就過）。改用**會成長的量**，再用一個**比率**當護欄。

### C. 指標體系規則（漏斗 + 護欄，先讀附錄 B）

- 指標要能串成一條**漏斗**：`原始量能 ×（功能層轉換率）＝ 北極星（價值已實現的量）`。各層不要互相撞名、語意重疊。
- 護欄分兩類：**正確性**（品質對不對）與**行為健康**（使用者行為健不健康）。
- 護欄要是**可量測、可優化的比率（0–100%）**，方向一致（「不下降」或「不上升」）；**不要用絕對門檻**（如「= 0」無法量測表現、無法迭代），**不要用高變異難監測的量**（如平均 session 時長 —— 會被長對話當離群值、還要定義 session 邊界）。
- **已經被技術解決的問題不要當 KPI**（例如「不截斷 / 截斷率」若已在工程上排除，就不該出現在護欄）。

### D. 簡報語言規則（這是給人看的，不是內部文件）

- **標題講白話、講這頁在說什麼**，不要用內部術語當標題（❌「機制 — 功能怎麼推動北極星」→ ✅「產品怎麼運作 — 從提問到使用者敢信」）。
- **專有名詞（北極星、護欄…）要在第一次出現的那頁先定義再使用**，不要假設聽眾懂。
- **不要赤裸裸地展現框架感**：不要把「STEP 01…07」這種教學框架標籤直接印在投影片上、也不要做一頁「框架總覽」。用**內容導向的章節名**（現況 / 使用者 / 成功定義 / 運作方式 / 驗證）。

### E. 視覺與排版建議（給後續排版/生成 SVG 用）

- **證據用大數字當主角**，不要堆成一段論文。每個區塊**項目 ≤ 3 條**，長統計改「一句重點 + 括註來源」。
- **每頁一條清楚的閱讀順序**（由上而下的分層標題 / 由左到右），別讓兩欄並列互搶、沒有主次。
- **留足呼吸感**：寧可少寫，不要塞滿；版面留白充足。

### F. 誠實規則

- **數字只在「驗證計畫」與必要處出現；「成功定義」那頁不放數字。**
- 沒有真實數據就標 `【需補充：要補什麼】`（如上線後流量、真實訪談金句、各指標基準值）；能依產品本質合理推斷的，標 `【待確認：推斷依據】`。**一律不要編造。**
- 任何門檻/區間數值都註明「**假設，上線後用真實數據校準**」。

### G. 抽取三層規則

到 `<輸入文件>` 找素材，依三層處理：(1) **找得到** → 直接用；(2) **找不到、但能依產品本質合理推斷** → 草擬一版並標 `【待確認：依據】`；(3) **完全沒依據** → 標 `【需補充】`，不要編造。

### H. 數據驗證與並行搜尋規則（適用於具備網路搜尋能力之 Agent）

若執行本模板的 AI Agent 具備工具調用或網路搜尋能力，在產出 `STEP 1 · 現況 / 問題` 頁面（簡報第一頁 Problem Exploration）時，**必須**強制執行以下元指令（meta-instructions）：
1. **分析提取痛點**：從 `<輸入文件>` 的 `spec.md` 中提取核心痛點與業務領域。
2. **設計 10 個搜尋查詢**：基於提取的核心痛點，設計 10 個具體的網路搜尋查詢（Search Queries），重點聚焦於市場規模、痛點工時、財務損失、錯誤率、行業調查或權威報告。
3. **並行啟動 10 個搜尋子代理人**：調用搜尋工具或並行啟動 10 個搜尋子代理人（parallel search subagents），分別執行上述查詢，蒐集具體定量統計、權威機構名稱（如 McKinsey、Bloomberg、WBCSD、Asana 等）與精確網址（URL）。
4. **融入實證數據至 Slide 1**：將蒐集到的真實數據與來源連結融入到 `STEP 1` 的「證據」中，完全替換任何推測或 `【需補充】` 標記。

> 輸出語言：繁體中文。每頁用 `## <章節名>` 開頭。few-shot 以下方「法規 RAG agent」當**示範**（學它的具體程度與語氣，不要照抄內容）。

## STEP 1 · 現況 / 問題（THE PROBLEM）

- **核心問句**：這是哪個領域、遇到什麼問題？問題的本質是什麼？為什麼非解不可？
- **填空骨架**：
  - 領域與問題一句話：______（標題要讓人一看就懂「誰、什麼問題」）
  - 證據（建議 2–3 個關鍵數字）：______（大數字主角 + 來源）
  - 問題的本質 / 根本原因：______（把現象翻成「為什麼會這樣」）
  - 為什麼非解不可：它導致 ______（成本 / 風險 / 代價）
- **抽取規則**：到 spec 找「問題定義 / Problem Statement / 痛點 / 現況不足」；plan 找促成此專案的動機。沒有量化證據就標 `【需補充：佐證數據】`。**這頁不要出現北極星。**
- **few-shot（法規 RAG agent）**：
  > 標題：法遵研究的根本困境 — 還沒有可信的 AI 助手
  > 證據：人工研究每週約 14.5 小時（LexisNexis）｜通用 AI 法律幻覺 58–88%（Stanford）｜全球逾 480 件 AI 假引註受罰
  > 本質：使用者「不敢信 → 凡事重新人工查核」；缺的不是 AI，是「快 + 可信（可追溯官方條文）」兼具的助手。
  > 非解不可：補上「可信」才能回收每年數百小時、避開高額裁罰。
- **輸出格式**：
  ```
  ## 現況 / 問題
  - 標題（白話）：【領域 + 問題】
  - 證據：【大數字1（來源）】｜【大數字2（來源）】｜【大數字3（來源）】
  - 問題的本質：【根本原因】
  - 為什麼非解不可：【代價/風險】
  ```

## STEP 2 · 使用者（THE USER）

- **核心問句**：誰、在什麼情境、有什麼還沒被滿足的痛點？
- **填空骨架**：
  - Persona：______（聚焦單一角色）
  - User Story：身為 ____，我想要 ____，以便 ____
  - 痛點收斂：______（≤ 2 條）
  - （選填）質性洞察 / 訪談金句：______
- **抽取規則**：到 spec 找「目標使用者 / persona / user story / 使用情境」。沒有訪談金句就標 `【需補充：訪談洞察】`。
- **few-shot（法規 RAG agent）**：
  > Persona：無專職法務的中小企業法遵承辦人員。
  > User Story：身為法遵承辦，我想快速查到某法規義務並看到條文出處，以便不必翻遍全國法規資料庫就能回報並自證可信。
  > 痛點：① 法源分散（法條/細則/函釋/FAQ 各自為政）② 通用 AI 會幻覺、無法追溯，法遵不敢用。
- **輸出格式**：
  ```
  ## 使用者
  - Persona：【…】
  - User Story：身為【…】，我想要【…】，以便【…】
  - 痛點：①【…】②【…】
  - 洞察/金句：【… 或 需補充】
  ```

## STEP 3 · 成功定義（DEFINITION OF SUCCESS，這頁不放數字）

- **核心問句**：相對現況，什麼若成為「常態」，提案就算成功？
- **填空骨架**：
  - 成功定義：要成功，必須讓 ______ 成為每一次（互動）的常態（質性、目標導向，**不放數字**）。
  - （建議）拆成 2–3 個並列的「成功條件」，正面陳述（不要用 before/after 把「已解決的技術問題」當現況）。
- **抽取規則**：呼應 STEP 1 的問題，從 spec 的目標/驗收標準提煉**質性**方向句。把任何數字移到驗證計畫。
- **few-shot（法規 RAG agent）**：
  > 成功定義：讓「答案完整 + 依據可一鍵追溯官方原文 + 對未涵蓋誠實標示」成為每一次問答的常態，而不是偶爾的驚喜。
- **輸出格式**：
  ```
  ## 成功定義（不放數字）
  - 一句話：必須讓【…】成為每一次的常態。
  - 成功條件：①【…】②【…】③【…】
  ```

## STEP 4 · 解方（THE SOLUTION）

- **核心問句**：這產品是什麼？關鍵流程 / 畫面為何？跟現有做法差在哪？技術可行嗎？
- **填空骨架**：
  - 一句話定義產品：______
  - 核心流程：步驟 1 → 2 → 3 …
  - 關鍵畫面：______（實機截圖 / mockup；優先放**有實際內容**的畫面，不要只放空的入口頁）
  - 競品差異：______（你跟現有做法本質差在哪）
  - 技術可行性重點：______
- **抽取規則**：到 plan 找「實作步驟 / 架構 / 流程 / 畫面」；spec 找競品或技術約束。
- **few-shot（法規 RAG agent）**：
  > 流程：輸入法遵問題 → SSE 串流分層答案 → 來源列 → 點腳註開引證抽屜外連官方原文 → 追問。
  > 畫面：放「實際問答畫面」與「引證抽屜（外連原文）」，不要只放空首頁。
  > 差異：相較通用 ChatGPT — 接地 2074 筆真實官方語料、每條可追溯、對未收錄誠實標示。
  > 技術：OpenAI Responses API + file_search 託管 RAG（不自建向量庫）。
- **輸出格式**：
  ```
  ## 解方
  - 一句話：【產品定義】
  - 核心流程：【1】→【2】→【3】
  - 關鍵畫面：【描述/截圖；放有實際內容的畫面】
  - 競品差異：【…】
  - 技術重點：【…】
  ```

## STEP 5 · 運作方式（HOW IT WORKS，先定義北極星，再講功能怎麼達到它）

- **核心問句**：功能透過什麼機制，一步步讓使用者真的感受到價值？
- **這頁是北極星第一次出現的地方 —— 先用一句話把它定義清楚，再講機制。**
- **填空骨架**：
  - **先講目標**：北極星 ＝ ______（一句話定義，且為「上線後承諾兌現的目標」；定義方式見附錄 A）
  - 流程怎麼走：步驟 1 → 2 → … → 兌現北極星
  - 每個機制的**因果**：因為這功能具有 ______（性質：可追溯 / 快速易懂 / 誠實兜底…），所以帶動 ______（北極星 / 量能 / 守住護欄的哪一塊）。**一定要講出「為什麼能帶動」，不要只列功能。**
- **抽取規則**：到 spec/plan 找「核心功能 / 解法」，把它的性質連到價值，講出因果。
- **few-shot（法規 RAG agent）**：
  > 先講目標：北極星 ＝ 每週「被查核的有效回答數」（附引證、且使用者真的展開查核）。
  > 機制 1（可追溯）：每條答案附可點引證 → 官方原文 ⇒ 直接推動北極星。
  > 機制 2（快速易懂）：串流即時、分層表格好讀懂 → 理解成本低 → 願意多問、回訪 ⇒ 帶動量能。
  > 機制 3（誠實兜底）：對「語料未涵蓋」誠實標示、不編造 ⇒ 守住正確性護欄、移除「不敢用」阻力。
- **輸出格式**：
  ```
  ## 運作方式（標題用白話，例：產品怎麼運作 — 從提問到使用者敢信）
  - 先講目標：北極星 ＝【一句話定義】
  - 流程：【1】→【2】→…→ 兌現北極星
  - 機制1：因為【性質】→【為什麼能帶動】⇒【北極星/量能/護欄】
  - 機制2：…
  - 機制3：…
  ```

## STEP 6 · 驗證計畫（VALIDATION，指標體系 + 護欄）

- **核心問句**：上線後怎麼證明成功？指標怎麼分層？有哪些護欄？
- **填空骨架（漏斗式，見附錄 B）**：
  - 北極星（產品線層）：______（同 STEP 5 定義，標明「上線後承諾兌現的目標」）
  - 功能層轉換率：______（原始量能 → 北極星 的轉換比率，可量測）
  - 量能指標：______（原始活躍量，**不要跟北極星撞名**）
  - 護欄 · 正確性：______（品質比率「不下降」，如 覆蓋率 / 忠實度；**不要用「= 0」**）
  - 護欄 · 行為健康：______（比率型、可監測、不怕離群值；**不要用平均 session 時長**）
- **抽取規則**：北極星同 STEP 5。指標要可量測。沒有就標 `【需補充：量化指標】`、門檻標「假設，上線校準」。
- **few-shot（法規 RAG agent）**：
  > 北極星：每週被查核的有效回答數（上線後承諾兌現的目標）。
  > 功能層：有效回答率 = 完整作答且每條附引證 ÷ 總回答（把總問答轉成有效回答的轉換率）。
  > 量能：每週總問答數 · 追問點擊數。
  > 護欄·正確性：依據覆蓋率不下降 · 忠實度不下降（答案忠於所引來源）。
  > 護欄·行為健康：來源查核率不下滑（點開引證/外連原文的回答比率）。
- **輸出格式**：
  ```
  ## 驗證計畫
  - 北極星（上線後目標）：【…】
  - 功能層轉換率：【…】
  - 量能指標：【…】
  - 護欄·正確性：【… 不下降】
  - 護欄·行為健康：【… 不下滑（比率型）】
  - 【需補充：各指標基準值與目標門檻】
  ```

---

## 附錄 A · 怎麼為「新產品」定一個好北極星

**好北極星的三個條件**：

1. **量到「價值被感受到」**（需求面 / 被採納），不是「系統做了什麼」（供給面）。
   - ❌「系統有附引證的回答數」（供給：你單方面附了，沒人看也算）
   - ✅「被使用者真的展開查核的有效回答數」（需求：只有使用者真的去查核才算）
2. **在現有限制下算得出來。** 兩個常見限制與解法：
   - **不知道使用者是誰** → 不要用「使用者數」。改用**事件/產出單位**（一則回答、一筆結果、一次完成）當計數單位 —— 你只要記「這一則有沒有發生那個價值動作」。
   - **每個 session 的操作數不一** → 不要用「每 session 平均」（長 session 會被當離群值灌歪）。改用**整週/整段期間的全域加總**：一個 session 做 10 次就貢獻 10 個單位、做 1 次就 1 個，每個單位權重一樣。
3. **門檻不能太鬆。** 別用「每人每週 ≥1 次」這種打勾就過的低門檻。改用**會成長的量（count）**當北極星，再搭一個**同行為的比率（rate）當護欄**（量要長、率別崩，是標準 count + rate 組合）。

**好用的命名公式**：`每週「被 <價值動作> 的有效 <產出單位> 數」`
- 例：被**查核**的有效**回答**數、被**採納**的有效**結果**數、被**完成**的有效**任務**數。

## 附錄 B · 指標體系與護欄設計原則

**1. 漏斗要串得起來、別撞名**

```
原始量能（總操作數）
   ×  功能層轉換率（變成「有效」的比率）
   =  北極星（價值已實現的量）
```
- 北極星是「量（count，要成長）」；功能層是「率（轉換率）」；量能是「原始活躍量」。三者語意不能重疊（例：北極星叫「有效回答數」，量能就別叫「有效問答數」，改叫「總問答數」）。

**2. 護欄分兩類、都要可量測可優化**

| 類別 | 管什麼 | 例 | 方向 |
|---|---|---|---|
| 正確性 | 答案/結果對不對 | 依據覆蓋率、忠實度（faithfulness） | 不下降 |
| 行為健康 | 使用者行為健不健康 | 來源查核率 / 有效互動率 | 不下滑 |

**3. 護欄的「不要」清單（都是這次踩過的坑）**

- ❌ **絕對門檻（如「依據空白 = 0」）** → 無法量測表現、無法迭代優化。改成**比率**（覆蓋率不下降）。
- ❌ **平均 session 時長** → 高變異、會被長對話當離群值、還要定義 session 邊界。改成**比率型**指標。
- ❌ **「任何點擊都算」的太鬆指標** → 好奇亂點就達標。改成**有語意的動作**（真的展開引證/外連原文）。
- ❌ **需要使用者識別的指標（回訪率…）** → 沒登入就算不出來。改成**事件/回答層級**的比率。
- ❌ **已被技術解決的問題（如「截斷率/不截斷」）** → 不是 KPI，別放護欄。

## 收尾（給 LLM）

各頁產出後：

1. 附一段 **3–5 句的口頭摘要**，用「問題 → 解方 → 怎麼驗證」串成一段話（**不要用指標開場**），方便提案開場時講。
2. 列出所有 `【需補充】` 與 `【待確認】` 清單，提醒使用者補齊 —— 特別是上線後真實數據、訪談金句、各指標基準值與門檻。
3. 提醒：所有門檻/區間都是**假設，上線後用真實數據校準**。
